What the Knowledge Graph Does

Every referral that enters the intake pipeline passes through the Knowledge Graph Validator — a deterministic rule engine that cross-checks extracted ICD-10 codes, HCPCS equipment codes, and clinical context against a structured set of CMS-sourced rules.

The KG does not use probability or language model inference for coding decisions. Each rule either fires or it does not, based on explicit conditions. When a rule fires, it returns the specific CMS guideline section that triggered it — creating a citation trail that survives audit.

In the Holloway referral, the KG ran 24 rules in under 100 milliseconds. Three fired: it rejected M23.611 (pre-surgical code on a post-surgical episode), confirmed S83.209A as the correct encounter code, and flagged a bariatric specification concern because the patient weighs 285 lbs — within 15 lbs of the 300 lb standard rollator capacity limit. The LLM resolved the ICD conflict; the KG validated it.

How the Knowledge Graph Was Built
📄
Step 1 — CMS Source Documents Acquired
The primary source is the CMS ICD-10-CM Official Guidelines for Coding and Reporting, FY2026 — a 144-page document updated annually by CMS. Secondary sources include CMS Local Coverage Determinations (LCD L33803 for walkers/rollators) and National Coverage Determinations (NCD) for DME equipment categories. These documents define the authoritative coding rules for every Medicare and workers comp DME claim in the US.
CMS ICD-10-CM FY2026 · LCD L33803 · CMS NCD
🧠
Step 2 — Rules Extracted and Structured into rules.json
Each relevant guideline section was extracted and converted into a structured JSON rule with explicit conditions, actions, source citations, confidence levels, and determinism flags. The 24 rules in rules.json cover the DME intake domain: post-surgical coding, pain classification, laterality, HCPCS coverage confirmation, equipment specification, workers comp requirements, and authorization. Each rule cites the specific CMS section (e.g., §I.C.19.a — Coding of Injuries) so every decision is traceable to its regulatory source.
24 rules · 7 categories · source-cited per rule
🕸
Step 3 — OWL Ontology Defines the Semantic Framework
The rule categories and relationships are structured using OWL (Web Ontology Language) — the W3C standard for representing knowledge domains. OWL provides formal class hierarchies (ICD code → post-surgical code → valid/invalid), property chains (post_surgical AND M23.xxx → REJECT), and disjointness axioms (M-codes and S-codes are disjoint in post-surgical context). This ensures the rules are not just a flat list of if/then statements but a connected semantic graph where relationships between code classes, clinical states, and equipment categories are formally represented.
W3C OWL 2 · RDF triples · class hierarchies
⚡
Step 4 — Python Rule Engine Executes the Graph
The knowledge_graph.py module loads rules.json at startup and runs all 24 rules against each episode's extracted fields. The engine evaluates rules in category order, collects all fired rules, and derives the overall validation status using a priority hierarchy: REJECTED > ESCALATED > FLAGGED > VALIDATED. The result includes every fired rule, its CMS source, confidence, and any replacement suggestion — all returned in under 150ms per episode. No database, no network call, no probability distribution.
Python · JSON rule cache · <150ms per episode
OWL — Web Ontology Language
Why OWL — Not Just Rules
A flat if/then rule list breaks when conditions overlap or contradict. OWL enforces formal logical consistency — class disjointness, property restrictions, and transitivity — so the knowledge graph remains coherent as rules are added or updated.
Class Hierarchy
ICD codes are typed as classes: PostSurgicalCode, TraumaticInjuryCode, DegenerativeCode. M23.xxx is a member of DegenerativeCode. S83.xxx is a member of TraumaticInjuryCode. These two classes are declared disjoint — a code cannot be both.
Property Chains
OWL property chains encode multi-condition rules: hasEpisodeContext(post_surgical) ∧ hasPrimaryCode(M23.xxx) → triggersRejection(DME-ICD-001). This is more expressive than a boolean AND — it can chain three or more conditions with formal semantics.
Inference-Ready
Because the ontology uses OWL-DL, a reasoner (Pellet, HermiT) can infer new facts. If a new ICD code is classified as a subclass of PostSurgicalCode, all rejection rules for that class apply automatically — without manually adding conditions for each new code.

Actions — What Each Outcome Means
CONFIRM
Code Validated
The rule positively confirms a code or field is correct. Contributes to overall VALIDATED status. No action required by staff.
Example: DME-ICD-002 fires — S83.209A confirmed as the correct post-ACL encounter code per §I.C.19.a. HCPCS E0143 confirmed covered under LCD L33803.
REJECT
Code Rejected
A coding error is definitively identified. The episode status becomes REJECTED. The agent provides the replacement code and CMS source. Billing cannot proceed until resolved.
Example: DME-ICD-001 fires — M23.611 rejected. Post-surgical episode must use S83.xxx not M23.xxx per §I.C.19.a. Replacement: S83.209A suggested.
FLAG
Review Required
A concern is detected that may not block dispatch but requires clinical or operational review. Episode status becomes FLAGGED. Outreach may be needed.
Example: DME-ICD-023 fires — Patient weight 285 lbs, within 15 lbs of E0143 300 lb limit. Flag sent to case manager to confirm bariatric specification before dispatch.
ESCALATE
Human Decision Required
Evidence is conflicting and confidence is below threshold. The agent will not self-correct. Episode is routed to clinical specialist for a binding decision.
Example: DME-ICD-005 fires — Laterality conflict detected (M25.361 right vs M25.362 left) at 71% confidence. Rule requires human review per §I.B.13.
Status Priority — The Guardrail That Makes This Deployable
Overall episode status is derived from all fired rules using a strict priority order: REJECTED > ESCALATED > FLAGGED > VALIDATED. If any rule fires a rejection, the episode is REJECTED regardless of confirmations. If any rule escalates and none reject, the episode is ESCALATED. This priority hierarchy means the agent never assigns VALIDATED to a episode that has a live coding concern — the safest outcome always wins.
